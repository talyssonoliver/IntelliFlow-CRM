# Production Credential Coverage — Remediation Spec (2026-06-07)

> Generated by the `credential-coverage-audit` multi-agent workflow (10 domain
> agents + synthesis) during the post-DB-rotation hardening session. Audited
> **73 credentials**: **7 Tier-0** (boot/decrypt-crash if absent), **10 Tier-1**
> (name-contract mismatches that silently break features), **38 Tier-2**
> (feature-gated, deferrable). This is the authoritative blueprint for modelling
> production secrets in HCP/Terraform and fixing the env-name contracts BEFORE
> the deferred full `terraform apply` (the 3 workers + observability).
>
> **Hard gate:** a full apply today would crash-boot the new workers and could
> null the live `SENTRY_DSN`. See the Pre-apply checklist at the bottom.
> Related: issue #312 (serverless `connection_limit`), ADR-065 (Vault-as-SSOT, TBD).

---

## Summary

A full `terraform apply` against the current plan would crash-boot all three new Railway workers (notifications-worker, ai-worker, workers) and the API service because seven Tier-0 secrets are absent from HCP entirely: `PRISMA_FIELD_ENCRYPTION_KEY`, `AI_AUDIT_SIGNING_KEY`, `LITELLM_MASTER_KEY`, `VAULT_TOKEN`, `VAULT_LOCAL_DEK_SECRET`, `GEMINI_API_KEY`, and `OPENROUTER_API_KEY`. An apply could also null out the live `SENTRY_DSN` value currently stored in `.env` but not yet in HCP, silently disabling all production error reporting. The audit found **7 Tier-0** (boot-crash or field-decrypt-crash if absent), **10 Tier-1** (name-contract mismatches that silently swallow config: email 5, SMS 3, DB pool 2), and **38 Tier-2** (feature-gated, defer until feature ships). Additionally, the `DATABASE_URL` is Tier-0 and already partially wired — it must be composed from the existing `supabase_db_pooler_host` HCP output at the transaction-pooler port 6543 (per PR #300) before any apply.

## Tier 0 — boot-required BEFORE any apply

All thirteen variables below must exist in HCP (or Railway native env store as noted) before running `terraform apply`. Wiring a service means adding the variable to the `shared_env_vars` map consumed by the Railway service resource, or to the `vercel_project_env` resource for web.

### 1. `PRISMA_FIELD_ENCRYPTION_KEY`

**Why boot-critical:** `packages/db/src/client.ts:16-21` hard-throws `[M4] PRISMA_FIELD_ENCRYPTION_KEY must be set in production` at module load when `NODE_ENV=production`. Every service that imports `@intelliflow/db` (api, ai-worker, workers, web via tRPC) will crash-boot without it.

**Value format:** 32-byte base64 string.

**Consumed by:** api, ai-worker, notifications-worker, workers, web.

```hcl
variable "prisma_field_encryption_key" {
  type      = string
  sensitive = true
}
```

Wire into `shared_env_vars` for Railway api, ai-worker, and all worker services. Wire into `vercel_project_env` for web.

### 3. `DATABASE_URL`

**Why boot-critical:** `packages/db/src/client.ts:106` uses a non-null assertion (`!`) at module load — any service importing `@intelliflow/db` will throw a runtime exception immediately if absent. Additionally `packages/validators/src/config.ts:158` requires a valid `postgresql://` URL with no default in `prodEnvSchema`.

**Value format:** `postgresql://postgres:<password>@<supabase_db_pooler_host>:6543/<db>?pgbouncer=true` (transaction pooler, per PR #300). Compose from existing HCP outputs `supabase_db_pooler_host` and `supabase_db_password` rather than storing the full URL as a static secret.

**Consumed by:** api, ai-worker, notifications-worker, workers (all db clients), web (via server components / tRPC).

```hcl
variable "database_url" {
  type      = string
  sensitive = true
}
# Preferred: compose in Terraform from existing HCP outputs
locals {
  database_url = "postgresql://postgres.${var.supabase_project_ref}:${var.supabase_db_password}@${var.supabase_db_pooler_host}:6543/postgres?pgbouncer=true"
}
```

Wire `local.database_url` into `shared_env_vars` for all Railway services and into `vercel_project_env` for web.

### 6. `AI_PROVIDER`

**Why boot-critical:** The `.env` default is `ollama` which is valid for local dev but is the wrong provider for production. `apps/ai-worker/src/config/ai.config.ts:135-142` throws in production when `AI_PROVIDER=mock`. While `ollama` does not throw, it causes all LLM calls to target a local Ollama instance that does not exist in Railway, resulting in silent failures on every AI job. Must be overridden to `litellm` (or `openrouter` once that PR lands) in the Railway env store.

**Consumed by:** api, ai-worker.

This is a non-secret plain variable. Set directly in Railway native env store per service rather than HCP. No Terraform variable block required, but document the override in the Railway service definition:

```hcl
env_vars = {
  AI_PROVIDER = "litellm"
}
```

## Vault vs HCP vs native env

### Recommendation

**Unblock now using HCP + Railway/Vercel native env stores.** Do not attempt to use HashiCorp Vault as the primary secret distribution mechanism for this apply cycle. Here is the rationale:

**What `VAULT_*` actually does in this codebase (per the audit):**

- `VAULT_ENABLED` (`encryption.ts:453`) gates whether `VaultKeyProvider` is instantiated. When absent or `false`, the code falls back to `EnvironmentKeyProvider`, which reads `PRISMA_FIELD_ENCRYPTION_KEY` directly from the process environment.
- `VAULT_TOKEN` and `VAULT_LOCAL_DEK_SECRET` (`encryption.ts:169,193`) serve as the **DEK (Data Encryption Key) seed** for field-level encryption. They are not a general secret distribution mechanism — they are a single application-specific key used to derive field encryption keys per version.
- `VAULT_ADDR` is the Vault cluster endpoint, only active when `VAULT_ENABLED=true`.

In summary: Vault in this codebase is an **optional alternative DEK provider**, not a secrets manager that distributes env vars. Using HCP Vault Secrets (the TF-variable store) for secret distribution is the correct model here, and it is already partially adopted (Supabase, Vercel, and Railway tokens are in HCP).

**Short-term (this apply cycle):**

1. Add all Tier-0 sensitive variables (`prisma_field_encryption_key`, `ai_audit_signing_key`, `litellm_master_key`, `redis_password`, `vault_token`, `vault_local_dek_secret`, `gemini_api_key`, `openrouter_api_key`, `sentry_dsn`) to HCP Vault Secrets as sensitive Terraform variables.
2. Set non-sensitive Tier-0 values (`redis_host`, `redis_port`, `ai_provider`, `litellm_base_url`) on Railway native env store directly, or as non-sensitive HCP vars.
3. Compose `DATABASE_URL` from existing HCP outputs (`supabase_db_pooler_host`, `supabase_db_password`) in a Terraform `local` block rather than storing the composed URL as a static secret.

**Long-term:**

Write an ADR (suggest ADR-065: Vault-as-SSOT) to decide whether to:
- Promote HashiCorp Vault Transit to primary DEK provider (`VAULT_ENABLED=true`) for field encryption, and
- Whether to use Vault Agent / VSO to inject env vars into Railway containers.

Until that ADR is approved and the Vault cluster is provisioned in production, `VAULT_ENABLED` should remain unset (or `false`), and `VAULT_LOCAL_DEK_SECRET` + `VAULT_TOKEN` should be set in HCP and injected by Terraform as standard env vars.
