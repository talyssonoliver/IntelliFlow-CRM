# Schema proposal — TenantMemoryPolicy + TenantAIConfig rate limits

**Date:** 2026-04-19 **Status:** PROPOSED — awaiting owner approval before
touching `schema.prisma` **Origin:** 2026-04-19 agent-system audit, gaps flagged
as P2.7 / P2.8

## Why this exists

The code-verified audit on 2026-04-19 surfaced two genuine gaps that require
Prisma schema changes:

1. **TenantMemoryPolicy** — conversation/memory retention is currently global
   and untyped. There is no per-tenant retention knob, so regulated tenants
   (GDPR right-to-erasure windows, HIPAA-ish deal-room separation, EU-only data
   residency) have to rely on ad-hoc app logic or manual DB pruning.

2. **Per-tenant rate limits** — `apps/ai-worker/src/config/ai.config.ts:114`
   exposes `rateLimitPerMinute` solely via the global `AI_RATE_LIMIT_PER_MINUTE`
   env var. A noisy tenant can exhaust the shared budget for everyone. There is
   no DB-backed override path.

Both require DB schema changes; per the standing "no DB drift without owner
approval" rule, this file proposes the diff without applying it.

## Proposed diff — `packages/db/prisma/schema.prisma`

### 1. New model: `TenantMemoryPolicy`

```prisma
/// Per-tenant retention policy for conversation / memory artifacts.
/// Controls what the background pruning cron deletes after N days.
/// Absent row = platform default (see ai.config.ts MEMORY_DEFAULT_RETENTION_DAYS).
model TenantMemoryPolicy {
  id                       String   @id @default(cuid())
  tenantId                 String   @unique
  /// Delete ConversationRecord + MessageRecord older than N days (null = never).
  conversationRetentionDays Int?
  /// Delete ChainVersion rows archived >= N days (null = keep forever).
  chainVersionRetentionDays Int?
  /// Delete AIMonitoringEvent rows older than N days (null = keep forever).
  monitoringEventRetentionDays Int?
  /// If true, PII-scrub conversation bodies at retention boundary rather than delete.
  scrubRatherThanDelete    Boolean  @default(false)
  /// Optional free-text justification surfaced in audit logs.
  reason                   String?
  createdAt                DateTime @default(now())
  updatedAt                DateTime @updatedAt

  tenant                   Tenant   @relation(fields: [tenantId], references: [id], onDelete: Cascade)

  @@index([tenantId])
  @@map("tenant_memory_policy")
}
```

Also add to `model Tenant`:

```prisma
  memoryPolicy   TenantMemoryPolicy?
```

### 2. Extend `TenantAIConfig` with per-tenant rate limits

```prisma
model TenantAIConfig {
  id                  String   @id @default(cuid())
  tenantId            String
  purpose             String
  tier                String
  temperature         Float?
  maxTokens           Int?
  // NEW — P2.8 per-tenant rate limit override.
  // null ⇒ fall back to AI_RATE_LIMIT_PER_MINUTE env var.
  rateLimitPerMinute  Int?
  createdAt           DateTime @default(now())
  updatedAt           DateTime @updatedAt
  tenant              Tenant   @relation(fields: [tenantId], references: [id], onDelete: Cascade)

  @@unique([tenantId, purpose])
  @@map("tenant_ai_config")
}
```

## Migration SQL — draft

> This SQL is **draft only**. The actual migration must be generated by
> `pnpm --filter @intelliflow/db exec prisma migrate dev --name tenant_memory_and_rate_limits`
> against a clean shadow DB so Prisma resolves column order, indexes, and
> constraint names against the engine's canonical form.

```sql
-- P2.7 — TenantMemoryPolicy
CREATE TABLE "tenant_memory_policy" (
  "id"                           TEXT NOT NULL,
  "tenantId"                     TEXT NOT NULL,
  "conversationRetentionDays"    INTEGER,
  "chainVersionRetentionDays"    INTEGER,
  "monitoringEventRetentionDays" INTEGER,
  "scrubRatherThanDelete"        BOOLEAN NOT NULL DEFAULT false,
  "reason"                       TEXT,
  "createdAt"                    TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP,
  "updatedAt"                    TIMESTAMP(3) NOT NULL,
  CONSTRAINT "tenant_memory_policy_pkey" PRIMARY KEY ("id")
);

CREATE UNIQUE INDEX "tenant_memory_policy_tenantId_key" ON "tenant_memory_policy"("tenantId");
CREATE INDEX "tenant_memory_policy_tenantId_idx" ON "tenant_memory_policy"("tenantId");

ALTER TABLE "tenant_memory_policy"
  ADD CONSTRAINT "tenant_memory_policy_tenantId_fkey"
  FOREIGN KEY ("tenantId") REFERENCES "Tenant"("id") ON DELETE CASCADE ON UPDATE CASCADE;

-- P2.8 — TenantAIConfig.rateLimitPerMinute
ALTER TABLE "tenant_ai_config" ADD COLUMN "rateLimitPerMinute" INTEGER;
```

Both migrations are **additive** and **non-destructive**: no existing rows are
touched, the new column on `tenant_ai_config` defaults to NULL which the app
treats as "use env var default".
